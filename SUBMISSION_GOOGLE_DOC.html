Project Overview

A production-ready, real-time Multiplayer Tic-Tac-Toe game built with a server-authoritative architecture using Nakama as the backend game server. All game logic runs on the server — clients only send intent; the server validates, applies, and broadcasts every state change.

Features Implemented

Core Requirements

  • Server-Authoritative Game Logic — Board state, turn management, and win detection all run on the Nakama server. Clients cannot manipulate game state.
  • Real-Time Multiplayer — WebSocket-based communication for instant, bidirectional game state updates between players.
  • Matchmaking System — find_match RPC automatically pairs players. Finds open matches first; creates a new one if none available.
  • Anti-Cheat — Server rejects out-of-turn moves, occupied-cell moves, and out-of-bounds positions.

Bonus Features

  • Concurrent Game Support — Multiple simultaneous matches with fully isolated game rooms.
  • Leaderboard System — Global ranking persisted in Nakama storage: Win +200 pts, Draw +25 pts, Loss −50 pts. Top 20 players displayed in-game.
  • Win Streak Tracking — Per-player streak counter, reset on loss or draw.
  • Timer-Based Game Mode — 30-second per-turn timer (selectable at match start). Server auto-forfeits on timeout — no client involvement needed.

Architecture

Server-Authoritative Flow

Client A                    Nakama Server                   Client B
   |                             |                             |
   |-- MOVE {position: 4} ----->|                             |
   |                             |-- validate turn order       |
   |                             |-- validate cell is empty    |
   |                             |-- apply move to board       |
   |                             |-- check for winner          |
   |<-- STATE_UPDATE -----------|-- STATE_UPDATE ------------->|
   |                             |                             |

OpCodes (Client ↔ Server Protocol)

CodeNameDirectionDescription
1MOVEClient → ServerPlayer sends move {position: 0–8}
2STATE_UPDATEServer → ClientFull game state broadcast after each valid move
3GAME_OVERServer → ClientGame result with winner / draw / reason
4MATCH_READYServer → ClientBoth players joined — game begins
5OPPONENT_LEFTServer → ClientOpponent disconnected — remaining player wins

RPC Endpoints (Backend API)

RPCPayloadResponse
find_match{mode: "classic" | "timed"}{matchId: string}
get_player_stats{}{wins, losses, draws, streak, score}
get_leaderboard{}{records: LeaderboardEntry[]}

Matchmaking Flow

  1. Client calls find_match RPC with chosen mode (classic or timed)
  2. Server lists open authoritative matches filtered by mode label
  3. If an open match exists (0–1 players), returns its match ID
  4. Otherwise, creates a new authoritative match
  5. Client joins via WebSocket socket
  6. When 2nd player joins, server broadcasts MATCH_READY and game begins

How to Run Locally

Prerequisites

  • Node.js v18+
  • Docker & Docker Compose

Step 1 — Start Nakama Backend

cd nakama
npm install
npm run build
docker-compose up -d

Nakama available at:

  • Game API: http://localhost:7350
  • Admin Console: http://localhost:7351 (admin / password)

Step 2 — Start Frontend

npm install
npm run dev

App runs at http://localhost:8080

Step 3 — Test Multiplayer

Open http://localhost:8080 in two separate browser windows (or one in incognito). Enter different nicknames, select the same mode, click Continue — both players are auto-matched.

Key Design Decisions

Why Nakama?

Nakama is purpose-built for game backends — it provides authoritative match handlers, built-in WebSocket transport, leaderboards, and persistent storage out of the box, making it the ideal server for a real-time multiplayer game.

Why Server-Authoritative?

Prevents cheating entirely. Clients only send intent (move position); the server validates and applies all state changes. Clients receive the authoritative state via broadcast — they cannot fake or replay moves.

Timed Mode Implementation

The server tracks turnDeadlineTick (the tick number when the turn expires). On every matchLoop tick, if tick >= turnDeadlineTick, the current player forfeits and the opponent wins — enforced entirely server-side with no client trust.

Leaderboard Persistence

Player stats (wins, losses, draws, streak, score) are stored in Nakama's storage engine with permissionWrite: 0 (server-only writes). The leaderboard uses Nakama's native incremental leaderboard — clients can only read, never write scores.
